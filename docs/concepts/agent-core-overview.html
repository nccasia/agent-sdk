01 — core idea

A Cartesian surface for thinking

The engine decouples what an assistant thinks about from how it progresses. Both are tuned independently — and a metacognition layer watches the whole plane.

02 — OY axis

Lobes — the context axis

“A lobe is a small thinking unit.” It organizes context and local behavior for a slice of the turn — and deliberately does not decide the whole execution plan.

The four-part lobe contract

  • stateReactive gates deciding whether the lobe activates this turn.
  • contextSelected information for the prompt & the Blackboard.
  • promptStable, slow, or volatile prompt blocks it contributes.
  • behaviorOptional executable work through the injected LLM/tool seam.

Typical lobes

memory_recall session_recall task_state skill_activate research synthesize cite · pinned filter · pinned format

Per-bot tuning uses flat weights to disable or bias a lobe's state nodes — kept in a namespace separate from flow_ so context tuning and flow tuning never leak.

04 — intent

Paths — the intent bridge

A path is the turn's recognized intent. It biases lobes (OY) and selects the flow (OX) — recognized every turn on free signals (lexical or structural), never an LLM.

Path → flow becomes a pure data dependency through the Blackboard: flows/ never imports paths, paths never import flows, and recognition is produced once, consumed by both axes. Carrying the full score vector opens the door to multi-intent flow adaptation — lobes already blend every recognized path's bias.

06 — long-running

Tasks — long-rail, autonomous execution

A task fires on a schedule, runs across many continuation fires (max_continuations), advances a bot_task_todos checklist — and must not lose the thread between fires. The trick: the engine writes its own working state to durable memory and reads it back.

The self-context loop / per fire

Steps 1 & 2 round-trip through context_entries (conversation scope, isolated server-side) — a coherent self-memory across arbitrary fires, with zero raw-chunk leakage.

landed · phase 1

The scratchpad convention

memory{ save, scope:"conversation",
  key:"decision_* / var_* / intent", ttl_days:N }

The task_execution lobe emits a pinned task_exec:scratchpad node; the ## Memory block surfaces it back on the next fire.

Seven capabilities, mapped to primitives

  • 1Self context-management — scratchpad notes survive the window reset.
  • 2Long-rail plan / DSL — the todo checklist is the execution rail.
  • 3Per-step context awareness — each rail step pulls only its lobe slice.
  • 4Error handling — mark blocked/failed, note it, let meta retry or escalate.
  • 5Report progress — a progress line per fire without ending the task.
  • 6Pause for human — request_human → waiting_for_human.
  • 7Redirect on the fly — a reply or fact rewrites the rail (add/skip/reorder todos).

07 — tool boundary

MCP — the internal tool boundary

One model-facing surface — the memory tool — over four scopes. The isolation guarantee is structural: the model picks a scope; the harness resolves the ref from injected identity.

The single model-facing tool

memory {
  "action": save | forget | recall,
  "scope": bot | user | channel | conversation,
  "key": snake_case_id,
  "value": <any>, "ttl_days": 7
}

save upserts in place · forget hard-deletes · recall reads one key or lists visible entries (only when a fact is truncated).

The isolation guarantee

The MCP contract strips any caller-supplied identity (tenant_id / bot_id / user_id / channel_id / conversation_id / scope_ref) before the request is built. A turn can only read/write the refs for its own identity — user A never sees user B's facts; channel X never leaks into channel Y.

08 — context-aware

Context-aware reasoning

An always-on ## Memory index is injected every turn so most recalls need zero tool calls — the answer is already in the prompt. When budgets bite, an adaptive builder scores what stays.

Four scopes — broad → specific

boteveryone, everywhere · admin-pinned
userone person, any channel
channelthe common case — a team in a shared channel
conversationone thread only

Rendered bot → user → channel → conversation so the most specific fact sits nearest the question and wins on conflict. Writes are update-in-place — same key overwrites, never a _v2 near-duplicate.

The adaptive context builder

Under context_strategy ∈ {adaptive, llm} the index returns post-cap structured rows. Each memory node is scored, not dumped:

Plus per-scope boosts: conversation > channel > user > bot. Budgets: INDEX_MAX_PER_SCOPE = 12, ~1200-char value budget; past it, entries degrade to hint-only with a “recall to read” note.

memory ≠ variables. Memory is what the bot chose to remember; context variables are authoritative facts the engine recomputes every turn — never invented or stale.
09 — meta-coordination

Metacognition — thinking about thinking

Always on since ENGINE 0.7.1. Two jobs: monitor the object-level engine, then regulate the next step. It never lets an LLM judge the pipeline.

observe the floor

default safe

Monitor and trace every decision — never change execution.

apply

Monitor, trace, and apply the allow-listed regulation actions.

METACOGNITION=observe

The kill switch — it kills mutation, not monitoring. Legacy off-tokens degrade to shadow mode, never to blindness.
